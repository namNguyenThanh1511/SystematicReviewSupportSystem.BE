## 2. 🚨 API Contract Issues (Gap Report)

### BLOCKER #1: Papers endpoint missing `abstract` and 10+ metadata fields

| Property | Detail |
|---|---|
| **Endpoint** | `GET /api/study-selection/{id}/papers` |
| **Severity** | **BLOCKER** |
| **What UI needs** | Full paper metadata: `abstract`, `journal`, `source`, `keywords`, `publicationType`, `volume`, `issue`, `pages`, `publisher`, `language`, `pdfUrl`, `conferenceName`, `conferenceLocation`, `journalIssn`, `url`, `publicationYearInt` |
| **What API provides** | Only 5 fields: `paperId`, `title`, `doi`, `authors`, `publicationYear` |
| **Why this is a problem** | The entire screening workflow is based on reading the abstract. Without `abstract`, the PaperViewer cannot display the core content. Keyword highlighting (the primary screening aid) is impossible. The metadata grid shows nothing. The PDF actions are broken without `pdfUrl`. |
| **Backend change** | Expand `PaperWithDecisionsResponse` to include all `PaperResponse` fields, or at minimum: `abstract`, `journal`, `source`, `keywords`, `publicationType`, `pdfUrl` |
| **Frontend workaround** | Fetch full paper details via a separate endpoint (e.g., `/api/papers/{id}`) when a paper is selected. This creates N+1 calls but unblocks the workflow. |

### BLOCKER #2: No reviewer name in screening decisions

| Property | Detail |
|---|---|
| **Endpoint** | `GET /api/study-selection/{id}/papers` → `decisions[].reviewerId` |
| **Severity** | **BLOCKER** |
| **What UI needs** | `reviewerName: string` for each decision (displayed as chips in PaperListItem) |
| **What API provides** | Only `reviewerId: string` (UUID) |
| **Why this is a problem** | Showing raw UUIDs instead of human names is unusable. Requires either N user lookups or name embedded in response. |
| **Backend change** | Add `reviewerName: string` (or `reviewerEmail`) to `ScreeningDecisionResponse` |
| **Frontend workaround** | Batch-fetch user profiles by IDs and cache in a Map. Show "Reviewer" as fallback until names load. |

### WARNING #1: Statistics endpoint counts only resolutions, not unanimous decisions

| Property | Detail |
|---|---|
| **Endpoint** | `GET /api/study-selection/{id}/statistics` |
| **Severity** | **WARNING** |
| **What UI needs** | `includedCount` = all papers with computed status "Included" (unanimous or resolved). Same for excluded. |
| **What API provides** | `includedCount` / `excludedCount` only count papers with a `ScreeningResolution` record. Unanimous reviewer decisions without formal resolution are NOT counted. |
| **Why this is a problem** | Stats in header/sidebar won't match what users see in the paper list. A paper showing green "Included" status in the list won't be counted in the stats bar. |
| **Backend change** | Count by computed status (Included/Excluded = status 1/2) not by resolution records only. |
| **Frontend workaround** | Compute stats client-side from the papers list instead of using the statistics endpoint. |

### WARNING #2: "Resolved" status (4) not handled in UI

| Property | Detail |
|---|---|
| **Endpoint** | `GET /api/study-selection/{id}/papers` → `status: 4` |
| **Severity** | **WARNING** |
| **What UI needs** | UI has 4 statuses: pending, included, excluded, conflicted |
| **What API provides** | 5 statuses: Pending(0), Included(1), Excluded(2), Conflict(3), Resolved(4) |
| **Why this is a problem** | Papers with status `Resolved(4)` have no UI representation and would fall through to "unknown". |
| **Frontend fix** | Map `Resolved(4)` → "included" or "excluded" based on `resolution.finalDecision`. If `finalDecision === 0` → "included", if `1` → "excluded". |

### WARNING #3: Decision immutability vs "Change Decision" button

| Property | Detail |
|---|---|
| **Endpoint** | `POST /api/study-selection/{id}/papers/{paperId}/decision` |
| **Severity** | **WARNING** |
| **What UI needs** | "Change decision" button exists in PaperViewer for already-decided papers |
| **What API provides** | Decisions are immutable. One decision per reviewer per paper. No update/delete endpoint. |
| **Why this is a problem** | The UI implies decisions can be changed, but clicking "Change decision" would result in a "Reviewer has already submitted a decision" error. |
| **Frontend fix** | Remove or disable the "Change decision" button. Show tooltip: "Decisions cannot be changed once submitted." |

### WARNING #4: No pagination on papers endpoint

| Property | Detail |
|---|---|
| **Endpoint** | `GET /api/study-selection/{id}/papers` |
| **Severity** | **WARNING** |
| **What UI needs** | For large reviews, paginated loading with server-side search/sort/filter |
| **What API provides** | Returns ALL papers in one flat array. No `pageNumber`, `pageSize`, `search`, `sortBy`, `status` query params. |
| **Why this is a problem** | For reviews with 500+ papers, loading all data at once will be slow and memory-intensive. |
| **Frontend workaround** | Client-side filtering/sorting/search is acceptable for now. Cache the full list with React Query `staleTime`. |
| **Backend recommendation** | Add optional query params: `?search=&status=&sortBy=&pageNumber=&pageSize=` |

### NICE_TO_HAVE #1: No resolver name in resolution response

| Property | Detail |

## 3. Frontend Integration Strategy

Given the BLOCKERs, the integration strategy is:

1. **For BLOCKER #1 (missing abstract)**: Compute stats client-side from the papers list. For paper details, if backend adds fields → use directly. If not → fetch full paper details on-demand when a paper is selected (requires a general `/api/papers/{id}` endpoint or use the existing PaperResponse from identification).

2. **For BLOCKER #2 (no reviewer name)**: Show `reviewerId` truncated as fallback. When a user lookup service is available, batch-resolve names.

3. **For WARNING #1 (stats mismatch)**: Compute stats client-side from the papers array using computed status mapping.

4. **For WARNING #2 (Resolved status)**: Map `Resolved(4)` to "included"/"excluded" based on `resolution.finalDecision`.

5. **For WARNING #3 (Change decision)**: Disable button with explanatory tooltip.

## 5. Frontend Risk Notes

1. **Race condition**: Two reviewers can submit decisions simultaneously for the same paper (by design). After submitting, re-fetch the paper list to see if a conflict was created.
2. **Optimistic updates**: Don't use optimistic updates for decisions — the backend may reject (already decided, process not InProgress). Always invalidate after mutation success.
3. **No undo**: Once a decision is submitted, it cannot be changed. Add a confirmation step before submission if the team wants safety.
4. **Performance**: No pagination means all papers load at once. Use `staleTime: 60_000` on React Query and client-side filtering. Consider `keepPreviousData` for smooth filtering transitions.
5. **Process lifecycle**: The workspace should check process status (NotStarted / InProgress / Completed) and show appropriate UI (e.g., "Start Screening" button, read-only mode for Completed).
